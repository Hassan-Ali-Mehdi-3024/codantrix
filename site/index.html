02 Process

Four phases. Each ends in a deploy.

The week-one deploy is small. The week-six deploy is real. We don't wait until the end to put code in front of users.

01 · Map

Discovery + scope

3 — 5 days

We map every decision point, data source, and failure mode in your workflow. Output: agent architecture spec, scoped SOW, signed decision doc before we touch code.

02 · Build

Spine first, agents later

2 — 3 weeks

Orchestration, memory, retrieval, tool calls. Deterministic where possible. We build the fail modes before the agents — most outages disappear.

03 · Wire

Agents in, evals running

2 — 3 weeks

Each agent has a specific scope, an eval suite that runs on every commit, a fail mode, and a metric. No "general assistant" agents.

04 · Ship

Production deploy + handover

1 week + 30 days

Deploy to your infra. Loom walkthrough. Runbook handed over. 30-day support window opens. After that: retainers or you take the wheel.

06 Questions

The ones we get asked.

What's the engagement model?

Fixed scope, fixed price, signed SOW before any work begins. 50/50 payment terms — 50% non-refundable upfront retainer, 50% on final delivery. IP transfers on final payment. No hourly billing, no T&M surprises. Change requests get their own SOW.

What happens if scope changes mid-engagement?

A change request is a new mini-SOW with its own price and timeline. We're transparent about what counts as a change versus what's inside scope. Most engagements ship without change requests because we spend the first week locking scope down hard.

Can you deploy to our infrastructure?

Yes. We deploy to your AWS, GCP, or Vercel. We can also deploy to ours during dev and migrate at handoff. We don't run a "platform" you depend on after the engagement closes — your code lives in your repo from day one.

Do you do enterprise compliance work (SOC 2, HIPAA)?

Not yet. If you need attested compliance for a regulated environment, we'll point you to a Layer 2 consultancy and step aside. Honest constraint; we don't pretend to be a fit when we're not.

Why are you "we" if it's a one-founder studio?

Because every engagement involves contractors — designers, frontend devs, sometimes a second backend pair. The engineering lead is always Hassan. The studio is small on purpose, but it's not solo when work is shipping.

What's your stack opinion?

Next.js for the frontend. Python or Node for the backend, depending on team fit. Claude or GPT-4 depending on the agent's job — we don't pick by brand, we pick by eval. LangGraph for orchestration when state matters; raw API calls when it doesn't. pgvector at small scale, Milvus when scale demands it.
